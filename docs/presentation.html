📊ข้อมูล: ERA5 รายวัน 1994–2023 (อุณหภูมิ + ความชื้นดิน) + ดัชนี ENSO & MJO → 20 features ต่อวันออกพยากรณ์ (มองย้อนหลังเท่านั้น กัน leakage)
🌡️นิยาม target: อุณหภูมิสูงสุดเฉลี่ยภูมิภาค (ถ่วง cos-lat) เกิน เปอร์เซ็นไทล์ 90 ต่อเนื่อง ≥ 3 วัน
🤖โมเดล: Logistic Regression & LightGBM (+ เวอร์ชัน recalibrate) — เบา รันบน CPU ได้
ระเบียบวิธี · การพิสูจน์ที่ "กันการโกง"

หัวใจคือ validation ที่เข้มงวด

⏳Rolling-origin CV: แบ่งข้อมูลตามเวลา (train อยู่ก่อน test เสมอ) + เว้นช่อง 49 วัน ระหว่างกัน → โมเดลห้าม "เห็นอนาคต"
⚖️2 baseline ที่ต้องเอาชนะ: seasonal climatology (ค่าเฉลี่ยตามวันในปี) และ persistence (ตอนนี้ร้อน→ร้อนต่อ)
📐เมตริก: Brier / BSS (สกิลของความน่าจะเป็น), AUC (การแยกแยะ), reliability (calibration)
🎲ความไม่แน่นอน: moving-block bootstrap (คุม autocorrelation) → ช่วงเชื่อมั่น 95% + คุม false discovery (BH-FDR)

ผลการทดลอง

📈 มาดูทีละกราฟ

แต่ละกราฟจะมีแผง "กราฟนี้บอกอะไร" อธิบายกำกับ

ผล 1 · สกิล + ความเชื่อมั่น

โมเดลเอาชนะ baseline ที่ระยะสั้น

📊 กราฟนี้บอกอะไร
  • แกนนอน = ล่วงหน้ากี่สัปดาห์ · แกนตั้ง = BSS (>0 คือเก่งกว่า climatology)
  • แต่ละเส้น = 1 โมเดล · แท่ง = ช่วงเชื่อมั่น 95% (จาก bootstrap)
  • ที่ lead 2 ทุกเส้น แท่งไม่แตะ 0 = ชนะอย่างมีนัยสำคัญ ; ไกลขึ้นสกิลจางลงตามธรรมชาติ
  • เส้นเขียว (logistic บาลานซ์+คาลิเบรต) อยู่เหนือเส้นประ persistence นานสุด
💡 สรุป: มีสกิลจริงและพิสูจน์ได้ที่ 2–4 สัปดาห์
ผล 2 · แยกตามสภาวะมหาสมุทร

เอลนีโญคือกุญแจ

📊 กราฟนี้บอกอะไร
  • แท่ง = BSS ในแต่ละสภาวะ ENSO · ตัวเลขกำกับ = base rate (ความถี่จริง)
  • ปีเอลนีโญ heatwave ถี่ ~0.45 (เกือบครึ่งวัน) · ลานีญา ~0.05
  • โมเดล แม่นสุดช่วงเอลนีโญ เพราะ climatology ทายต่ำไป โมเดลที่รู้ ENSO เลยชนะเยอะ
  • ลานีญา (เหตุการณ์น้อย) บางโมเดลทำได้ไม่ดี — สมเหตุสมผล
💡 สรุป: ปรับความเข้าใจเดิม — "ปีเอลนีโญไม่ได้อ่อน แต่กลับเป็นช่วงที่โมเดลเก่งสุด"
ผล 3 · คุณภาพความน่าจะเป็น

ทำไม LightGBM ต้อง recalibrate

📊 กราฟนี้บอกอะไร
  • เทียบ 3 วิธี: raw (ดิบ) / Platt / Isotonic · แกนตั้ง = BSS
  • เส้น raw (เทา) ติดลบทุก lead = ความน่าจะเป็น "เฟ้อ/มั่นใจเกินจริง"
  • Platt และ Isotonic ดึงกลับมาเป็นบวก ได้ทั้งคู่
💡 สรุป: boosting แยกแยะเก่งแต่ calibrate แย่ → ต้อง recalibrate ก่อนใช้เสมอ
ผล 4 · ข้อค้นพบหลัก

"น้อยแต่ดี" — ตัดฟีเจอร์ แล้วแม่นขึ้น

📊 กราฟนี้บอกอะไร
  • เทียบจำนวนฟีเจอร์: full(20) / no_thermal(16) / lean(10) / core(5)
  • เส้นเทา full(20) อยู่ล่างสุดทุกจุด และหลุดต่ำกว่า 0 ที่ lead 6
  • ยิ่งตัดฟีเจอร์ซ้ำซ้อน เส้นยิ่งสูงขึ้น — core(5) ดีสุดที่ lead ยาว
💡 สรุป: linear model overfit ฟีเจอร์ที่ซ้ำซ้อน → ความเรียบง่ายช่วยให้ generalize ดีขึ้น
ผล 5 · regularization vs feature selection

ตัดฟีเจอร์ ชนะ การ regularize

📊 กราฟนี้บอกอะไร
  • เทียบ: full / full+L2 แรง / core / core+คาลิเบรต / lean+คาลิเบรต
  • L2 แรง (ม่วง) ช่วยนิดหน่อย แต่ ยังสู้ core (ส้ม) ไม่ได้
  • เอา core ไป recalibrate (เขียว/น้ำเงิน) กลับ แย่ลง — เพราะ logistic calibrate ดีอยู่แล้ว
💡 สรุป: ผู้ชนะคือ core 5 ฟีเจอร์ธรรมดา ; calibration ช่วยแค่ boosting ไม่ใช่ logistic
ผล 6 · ฟีเจอร์ไหนสำคัญ

อะไรขับเคลื่อนการทำนาย

📊 กราฟนี้บอกอะไร
  • แท่งยาว = สลับค่าฟีเจอร์นั้นแล้วสกิล "ตกมาก" = สำคัญมาก
  • ฝั่ง Logistic: ฤดูกาล (doy) + ความชื้นดิน (sm) นำ → ยืนยัน land–atmosphere coupling
  • ฝั่ง LightGBM ดิบ: แท่ง error กว้าง (เพราะ miscalibration — ตรงกับผล 3)
💡 สรุป: 3 วัตถุดิบหลัก = ฤดูกาล + เอลนีโญ + ความชื้นดิน (ตรงกับชุด core)
