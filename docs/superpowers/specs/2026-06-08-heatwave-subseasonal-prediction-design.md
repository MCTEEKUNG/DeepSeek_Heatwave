# ดีไซน์: การทำนายความน่าจะเป็นของคลื่นความร้อนล่วงหน้าระดับ Sub-seasonal

วันที่: 2026-06-08
สถานะ: อนุมัติดีไซน์แล้ว / ผ่านการทดสอบดึงข้อมูล (de-risk) แล้ว
บริบท: งานวิจัย / วิทยานิพนธ์

---

## 1. คำถามวิจัยและเป้าหมาย

ทำนาย **ความน่าจะเป็น (probability)** ของการเกิดคลื่นความร้อนในพื้นที่ประเทศไทย / เอเชียตะวันออกเฉียงใต้
ล่วงหน้า **2–6 สัปดาห์ (sub-seasonal)** เพื่อเป็นข้อมูลสนับสนุนการเตือนภัยล่วงหน้า ลดอัตราการเสียชีวิต
และช่วยเหลือประชาชนได้มากที่สุด

**ขอบเขตงานวิทยานิพนธ์ (core):** โมเดล + การพิสูจน์สกิลการทำนาย
**อนาคต (Phase 2, นอกขอบเขตหลัก):** เว็บแอป / แอปมือถือสำหรับแสดงผลและแจ้งเตือน — ออกแบบให้ส่วน output รองรับไว้ แต่ยังไม่สร้าง

---

## 2. นิยามตัวแปรเป้าหมาย (Target)

- **คลื่นความร้อน** = Tmax รายวัน เกินเปอร์เซ็นไทล์ที่ **90** (และทดลองเปรียบเทียบที่ 95)
  ของภูมิอากาศฐาน โดยฐานคำนวณแบบ moving window (เช่น ±15 วันรอบวันปฏิทินนั้น)
  และต้องเกิด **ติดต่อกัน ≥ 3 วัน**
- แปลงเป็นปัญหา **classification เชิงความน่าจะเป็น**: ในสัปดาห์เป้าหมายที่ lead time 2/3/4/5/6 สัปดาห์
  จะมีคลื่นความร้อนเกิดขึ้นหรือไม่ (0/1) → โมเดลให้ผลเป็น probability

## 3. ข้อมูล (Data)

| ชนิด | แหล่ง | บทบาท |
|------|-------|-------|
| Tmax รายวัน | ERA5 `derived-era5-single-levels-daily-statistics` (`daily_maximum` ของ `2m_temperature`) | สร้าง target |
| Soil moisture | ERA5 volumetric soil water | **feature หลัก** — ตัวขับเคลื่อน predictability ที่แข็งแรงที่สุด<br>รอบแรกใช้ **ชั้น 1 (0-7cm) + ชั้น 3 (28-100cm)** (ดึง 1 ตัวแปร/คำขอ เพราะ 4 ชั้นพร้อมกันชน cost limit ของ CDS) ; เพิ่มชั้น 2/4 ภายหลังได้ |
| ดัชนีบรรยากาศ/มหาสมุทร | MJO RMM (BoM), Niño3.4/ENSO (NOAA PSL) | features (ตัวแปรเปลี่ยนช้า = memory ของระบบ) — **ดึง+แปลงเป็น CSV แล้ว** |
| (ภายหลัง) SST, geopotential/blocking | ERA5 | features เพิ่มเติมเมื่อ pipeline รอบแรกครบ |
| NDVI (ทางเลือก) | ดาวเทียม (MODIS/AVHRR) | **การทดลองเสริม (ablation)** ท้ายเล่ม ไม่อยู่ใน core model |

**ขอบเขตข้อมูลรอบแรก:** กรอบไทย, 30 ปี (1994–2023), **เฉพาะเดือน ม.ค.–ก.ค.** (ครอบฤดูร้อน ก.พ.–มิ.ย. + buffer)
เพื่อลดเวลาดาวน์โหลด ~ครึ่ง ; ขยายเป็นทั้งปีภายหลังได้ (สคริปต์ resume + ตรวจไฟล์อัตโนมัติ)
*หมายเหตุ edge effect:* เกณฑ์ percentile ของวันต้นเดือน ม.ค. จะใช้หน้าต่าง ±15 วันแบบไม่เต็ม (ขาดปลาย ธ.ค.) — ไม่กระทบแกนฤดูร้อน (มี.ค.–พ.ค. หน้าต่างเต็ม)

หมายเหตุเชิงวิทยาศาสตร์ (อ้างอิงงานวิจัย): Tmax อย่างเดียวไม่พอสำหรับการทำนายล่วงหน้า
soil moisture เป็นตัวทำนายระดับ sub-seasonal ที่แข็งแรงที่สุดผ่าน land–atmosphere coupling
ส่วน NDVI ส่วนใหญ่เป็น "ผลลัพธ์" ของ soil moisture+อุณหภูมิ จึงเสี่ยงซ้ำซ้อน → จัดเป็นการทดลองเสริม

**กลยุทธ์กัน limit:** ใช้ชุด daily statistics (ค่าสูงสุดรายวันมาตรงๆ), แบ่งดึง**ทีละปี**, กรอบเฉพาะไทย
ทดสอบแล้ว: 1 เดือน ≈ 0.16 MB → ชุดเต็มหลายสิบปี รวมไม่กี่ร้อย MB

## 4. โมเดล (ไล่จากง่ายไปยาก)

1. **Baselines (ต้องเอาชนะให้ได้):** climatology (ความถี่เฉลี่ยในอดีต) + persistence
2. **Logistic Regression** — โมเดลหลักที่อธิบายได้
3. **Gradient Boosting / XGBoost** — เพิ่มความแม่นยำ + วิเคราะห์ feature importance

## 5. การวัดผล (เน้นเชิงความน่าจะเป็น)

- **Brier Score / Brier Skill Score** (เทียบ climatology)
- **ROC-AUC**
- **Reliability diagram** (ความน่าเชื่อถือของ probability)
- นิยามความสำเร็จ = **ชนะ baseline climatology + persistence** (ไม่ใช่ accuracy สูงๆ
  เพราะ sub-seasonal เป็นช่วงที่ทำนายยากโดยธรรมชาติ — สกิลพอประมาณถือว่าโอเคถ้าระเบียบวิธีถูกต้อง)

## 6. กับดักสำคัญที่ต้องป้องกัน

- **Temporal leakage:** ข้อมูลอนุกรมเวลามี autocorrelation สูง — **ห้ามใช้ random CV**
  ต้องใช้ **blocked / rolling-origin time-series CV** โดยมี **ช่องว่าง (gap)** ระหว่างชุด train
  กับสัปดาห์เป้าหมาย เพื่อกันการรั่วของข้อมูลอนาคต

- **นโยบายหน่วย (Units policy) — กัน Kelvin/Celsius ปนกัน:**
  หน่วยมาตรฐานภายใน = **°C**. ทุกตัวแปรอุณหภูมิต้องผ่าน "ประตูเดียว" (`scripts/units_utils.py`)
  ที่แปลงตาม attribute `units` (ไม่เดาจากค่า) **ก่อนรวม/นำเข้าโมเดลเสมอ**.
  ตามด้วยด่านตรวจช่วงค่าที่ fail loudly (เช่น −90..65 °C) เพื่อดักกรณีลืมแปลง.
  หมายเหตุ: ERA5 ส่ง `2m_temperature` และ `sea_surface_temperature` มาเป็น **Kelvin** ทั้งคู่.
  อันตรายที่เงียบที่สุด = นำ 2 แหล่งคนละหน่วยมาต่อในคอลัมน์เดียว → ต้องแปลงที่ประตูทางเข้าก่อนเสมอ.

## 7. ลำดับการสร้าง (Milestones) — เรียงตามความเสี่ยง

1. **[เสร็จแล้ว ✅] ดึงข้อมูล ERA5 (de-risk):** ทดสอบ CDS API ก้อนจิ๋วผ่านแล้ว (เม.ย. 2023, Tmax, กรอบไทย)
2. ขยายการดึง Tmax เต็มช่วงเวลา (แบ่งทีละปี) + ดึง soil moisture
3. คำนวณ target (คลื่นความร้อนตามนิยาม percentile + ≥3 วัน)
4. รวบรวม/จัดดัชนี features ให้ตรง lead time
5. สร้าง baselines + กรอบการวัดผล (leakage-safe CV)
6. เทรน Logistic Regression → XGBoost
7. ประเมินผล + กราฟ (reliability, ROC) + เขียนสรุป
8. (ทางเลือก) ทดลองเสริม NDVI / บันทึก output รองรับ Phase 2
