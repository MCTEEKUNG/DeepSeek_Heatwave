1.การแบ่งข้อมูล — Train/Test แบบอนุกรมเวลา (ไม่ใช่ random)

ทำไม: ข้อมูลรายวันมี autocorrelation สูง ถ้าใช้ random split โมเดลจะ "เห็นอนาคต" ผ่านวันที่อยู่ติดกัน → สกิลเฟ้อ (leakage) การแบ่งสุ่ม 70/15/15 ใช้ไม่ได้กับงานนี้

วิธีที่ออกแบบ: RollingOriginCV — train อยู่ "ก่อน" test เสมอ + เว้น embargo gap = 42 วัน (~6 สัปดาห์) เพื่อตัดการรั่วข้าม lead time สูงสุด + หน้าต่างเป้าหมาย

รันสดในรอบนี้ python scripts/cv.py

# 5 ปีจำลอง, 4 folds — train มาก่อน test, gap=42 ทุก fold
fold 1: train [   0..1670] gap=42 test [1713..1740]
fold 4: train [   0..1754] gap=42 test [1797..1824]
[OK] ทุก fold: train มาก่อน test, มี gap เพียงพอ, ไม่ทับกัน
✅ ยืนยัน: ไม่มี temporal leakage — โครงสร้างการแบ่งข้อมูลถูกต้องตามนิยาม

2.Metric ที่เหมาะสม — ทำไม "Accuracy" จึงห้ามใช้

ทำไม: คลื่นความร้อนเป็นเหตุการณ์หายาก (base rate ~11–26%) โมเดลที่ทาย "ไม่เกิด" ตลอดก็ได้ accuracy สูงลวงตา งานนี้ต้องการ ความน่าจะเป็นที่ calibrate ดี จึงวัดด้วย Brier / BSS / AUC / reliability
Brier
ค่าเฉลี่ย (p−y)² · ต่ำ=ดี
BSS
เทียบ baseline · >0=ชนะ
AUC
การแยกแยะเหตุการณ์
Reliability
ทาย 30% เกิดจริง ~30%?

นิยามความสำเร็จ: ต้อง BSS > 0 เทียบ climatology และ AUC > 0.5 (แยกแยะได้จริง) — ดูทั้งสองคู่กัน เพราะ BSS บวกอย่างเดียวอาจเป็น calibration artefact (ดู lead 6 ในกล่องสถานะด้านบน)

รันสดในรอบนี้ python scripts/evaluate.py — ตรวจ metric กับค่าที่รู้คำตอบ:

พยากรณ์ climatology: bss≈0 (เท่า baseline)            ✅
พยากรณ์มั่ว       : auc≈0.48 (สุ่ม), bss=−2.63        ✅

8.Baseline Comparison + นัยสำคัญทางสถิติ

ทำไม: AI ที่ "เก่ง" ต้องชนะวิธีง่าย ๆ อย่างมีนัยสำคัญ ไม่ใช่ดีขึ้นนิดเดียวแบบบังเอิญ
  • Seasonal climatology — ความถี่ในอดีตตามวันในปฏิทิน (เรียนจาก train เท่านั้น)
  • Conditional persistence — P(เกิด | สถานะ ณ วันออกพยากรณ์) แทนการทาย 0/1 ดิบ

ทดสอบด้วย paired bootstrap + Benjamini-Hochberg FDR (กัน false positive จากการเทียบหลายคู่): ในเฟสวิจัย (§4) ทุก lead 2–6 ชนะทั้ง climatology และ persistence ที่ q < 0.05 ในโหมด OOS (§6) overall CI [+0.065, +0.232] ไม่คร่อม 0 → ชนะ climatology อย่างมีนัยสำคัญ CV OOS

✅ ชนะ baseline ที่แข็งแรงอย่างมีนัยสำคัญ — ส่วนต่างไม่ใช่ความบังเอิญ

9.Explainability — โมเดลตัดสินใจเพราะอะไร

10.User Acceptance Test (UAT)

ทำไม: โมเดลแม่นแต่คนใช้ไม่เข้าใจ/ตื่นตระหนกผิด = ยังไม่พร้อม ต้องสื่อสารความน่าจะเป็น + ความไม่แน่นอน + ขอบเขตที่เชื่อถือได้ ให้ชัด
  • docs/project-dashboard.html — แดชบอร์ดผลพยากรณ์รายจังหวัด + แผนที่ความร้อน
  • docs/architecture-explainer.html — อธิบายสถาปัตยกรรมระบบ
  • เปิดให้เห็นเฉพาะ lead 2–4 และ lead 4 ติดป้าย "Weakening skill" — ออกแบบมา เพื่อไม่ให้ผู้ใช้ตีความสัญญาณระยะไกลที่สกิลต่ำเกินจริง (กันความตื่นตระหนกโดยมิใช่เหตุ)
✅ การจำกัด horizon + ป้ายกำกับสกิล คือการนำ "ผลการทดสอบ §6" มาออกแบบ UX อย่างรับผิดชอบ
